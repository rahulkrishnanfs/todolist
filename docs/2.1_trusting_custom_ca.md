# Trusting a Custom CA Certificate (macOS, Linux, Docker)

## Overview

When running a local HTTPS server (e.g., a Go service), you may use a self-signed Certificate Authority (CA).  
To avoid TLS errors like:

- `NET::ERR_CERT_AUTHORITY_INVALID`
- `x509: certificate signed by unknown authority`

You must add your CA certificate (`ca.crt`) to the system trust store.

---

## macOS Setup

Use the following command to trust your CA system-wide:

```bash
sudo security add-trusted-cert -d -r trustRoot \
  -k /Library/Keychains/System.keychain \
  todolist/secrets/ca.crt

```

### What this does:

- Adds your CA as a trusted root certificate
- Stores it in the System Keychain
- Allows macOS and browsers to trust certificates signed by this CA

---

## Linux (Ubuntu/Debian)

### Step 1: Copy the certificate

```bash
sudo cp ca.crt /usr/local/share/ca-certificates/

```

### Step 2: Update the trust store

```bash
sudo update-ca-certificates

```

### Result:

- System tools like `curl`, `wget`, and Go HTTP clients will trust your CA

---

## Docker Setup

If your application runs inside Docker, you must also trust the CA inside the container.

### Example Dockerfile:

```dockerfile
COPY ca.crt /usr/local/share/ca-certificates/
RUN update-ca-certificates

```

### Important:

- Host trust does not apply to containers
- Each container must explicitly include the CA

---

## Browser Considerations

Even after installing the CA:

- Restart your browser
- If issues persist, manually import the certificate:
  - Chrome → Settings → Privacy & Security → Manage Certificates

---

## Common TLS Issues

### 1. Subject Alternative Name (SAN) Missing

Your certificate must include the hostname you're using.

Example:

```text
subjectAltName = DNS:localhost, IP:127.0.0.1

```

If missing, you'll see errors even if the CA is trusted.

---

### 2. Hostname Mismatch

Ensure:

- You access the server using the same hostname in the certificate
- Example:
  - Cert has `localhost` → Access via `https://localhost`
  - Not `https://127.0.0.1`

---

### 3. Remote vs Local Server

If your server runs remotely:

- Add the remote domain/IP in SAN
- Example:

```text
subjectAltName = DNS:myserver.com, IP:192.168.1.10

```

---

## Quick Checklist

- CA certificate installed on system
- Server certificate signed by CA
- SAN includes correct hostname/IP
- Client uses matching hostname
- Docker containers updated (if applicable)

---
